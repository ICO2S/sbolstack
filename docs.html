1. API Overview

The SBOL Stack provides a RESTful API to encapsulate the underlying triplestore, providing semantic access to the SBOL data model.

The API can be accessed using any standard HTTP client library. Additionally, native libraries encapsulating the stack API are available for Java and JavaScript.

Each instance of the SBOL Stack can provide an unlimited number of independent stores. Stores can be created using the /store/create endpoint, and then any further API call can be directed to a particular store by prepending /store/store_name/ to the URI. If no store is specified, the default store listed in the configuration will be used.

3. Installation

The SBOL Stack is deployed using Docker. Docker is a platform that allows pre-configured applications to be deployed anywhere, allowing the SBOL Stack to be installed as a single image with no prerequisites other than Docker itself.


From Prebuilt Image

First, install Docker. The installation instructions will vary depending on your operating system. For Ubuntu, the command is simply:

sudo apt-get install docker.io

Equivalent Docker installation instructions can be found for Mac and Windows.

There are two Docker images provided; an SBOL Stack container, and a SynBioHub container.

Starting a Stack container

# For the first run, create an emtpy data directory:
mkdir -p $HOME/docker/stack

# Download, install and run the SBOL Stack:
docker run \
    -v $HOME/docker/stack:/mnt/data \
    -p 9090:9090 \
    --name sbolstack -d keithf/sbolstack:v1.2.2d

Docker images are 'immutable'; once a given container is shut down, all modified data vanishes. Therefore, any persistent data needs to be located outside of the container, and mounted each time a container is booted. In the example above, we initially create an empty 'docker/stack' directory. This directory is then mounted inside the Docker container once it starts. Any parts that are uploaded to the Stack are stored within the 'docker/stack' directory, so you should ensure that this location is backed up appropriately

The '-p' option in the above command exposes port 9090 from the Stack container to 9090 on the host machine. Therefore, the SBOL Stack will now be available at http://localhost:9090 if you are running Docker on Linux, or at the address shown by docker-machine ip if you are running Docker on OSX or Windows. The Stack can be used immediately with any of the client libraries or the RESTful API.

Please note that you may need to wait for up to a minute before the SBOL Stack becomes become available on port 9090.

It is possible to start and stop the service as follows:

docker start sbolstack
docker stop sbolstack

If you no longer require the container, it can be deleted in the following way:

docker rm -v sbolstack

Note that any data stored in $HOME/docker/stack will not be deleted; you will need to delete the directory manually if you need to. Therefore, the docker rm operation is extremely useful when upgrading to a new Stack image version, since your database and configuration files remain intact.


Starting the SynBioHub container

If you wish to run your own instance of SynBioHub, you can do so via another Docker image. Here, we assume that you will be running the SynBioHub container on the same host as the SBOL Stack container described previously.

# For the first run, create an emtpy data directory:
mkdir -p $HOME/docker/synbiohub

# Download, install and run the SynBioHub:
docker run \
    -v $HOME/docker/synbiohub:/mnt/data \
    -p 7777:7777 \
    --link sbolstack:sbolstack \
    --name synbiohub -d keithf/synbiohub:v1.1.1

The Docker 'run' command for the SynBioHub is very similar to that of the SBOL Stack. However, note the --link option. This command line option allows the SynBioHub container to communicate with the SBOL Stack container. It is important to specify the same name as was used to create the SBOL Stack container in order for the link to work (in this case, we used sbolstack).

Once the SynBioHub container has started, you should be able to connect to the hub via the link: http://localhost:7777. The first time you run the SynBioHub, you will see a setup page, which asks you for the URL to the SBOL Stack that you wish to connect to. You should enter: http://sbolstack:9090 here. Again, this is the name of the SBOL Stack Docker container that we created in the previous step. If you used a different container name, you should modify the URL accordingly.

Please note that you may need to wait for up to a minute before the SynBioHub Web pages become available.

If you want to run the SynBioHub container on a different host machine, then the --link option is not required. In this case, when configuring the SynBioHub via its 'setup' page, you will need to enter the network hostname of the machine that is running the SBOL Stack container into the URL text box. The startup command for the SynBioHub is as follows:

docker run \
    -v $HOME/docker/synbiohub:/mnt/data \
    -p 7777:7777 \
    --name synbiohub -d keithf/synbiohub:v1.1.1



[Optional] Building the Docker images Source (Linux)

It is possible to build (and customise) your own Docker images. First, install Docker and git:

sudo apt-get install docker.io git



Building the SBOL Stack Docker image

First, clone the SBOL Stack docker repository:

git clone https://github.com/ICO2S/sbolstack-docker.git
cd sbolstack-docker/basic-image

You may want to change the default store name, users, and permissions in the configuration file. Then, build and run the Stack using Docker:

docker build -t {your-docker-user}/sbolstack:{your-version-tag} .

The image can then be run as follows:

docker run \
    -v $HOME/docker/stack:/mnt/data \
    -p 9090:9090 \
    --name sbolstack -d {your-docker-user}/sbolstack:{your-version-tag}

As with the prebuilt image, the SBOL Stack will now be available at http://localhost:9090. The Stack can be used immediately with any of the client libraries or the RESTful API.




Building the SynBioHub Docker image

Building the Docker image for SynBioHub is performed in a similar fashion. First, clone the SBOL Stack docker repository:

git clone https://github.com/ICO2S/synbiohub-docker.git
cd synbiohub-docker

Then:

docker build -t {your-docker-user}/sbolstack:{your-version-tag} .

Finally, a SynBioHub container can be created and started with the following command:

docker run \
    -v $HOME/docker/synbiohub:/mnt/data \
    -p 7777:7777 \
    --link sbolstack:sbolstack \
    --name synbiohub -d {your-docker-user}/synbiohub:{your-version-tag}


[Optional] Installation without Docker

It may be useful to install the SBOL Stack without Docker, in order to customise the installation and/or modify the SBOL Stack source code.

The prerequisites for the SBOL Stack are:

  • OpenJDK 8 or a compatible JVM
  • Apache Tomcat 8
  • OpenRDF Sesame version 4.x.x
  • node.js version 4.x.x with npm

On Ubuntu 16.04, the dependencies can be installed from aptitude:

sudo apt-get install git nodejs openjdk-8-jre-headless npm curl
sudo update-alternatives --install /usr/bin/node node /usr/bin/nodejs 10

The remaining dependencies, Apache Tomcat and OpenRDF Sesame, can be obtained using curl:

curl -L http://mirror.catn.com/pub/apache/tomcat/tomcat-8/v8.0.35/bin/apache-tomcat-8.0.35.tar.gz | tar xzf -
curl -L https://sourceforge.net/projects/sesame/files/Sesame%204/4.1.2/openrdf-sesame-4.1.2-sdk.tar.gz/download | tar xzf -
sudo cp -r apache-tomcat-8.0.35 /opt/

The OpenRDF Sesame WAR files then need to be placed in /opt/apache-tomcat-8.0.35/webapps/. Sesame provides two Web applications: openrdf-sesame and openrdf-workbench. While only openrdf-sesame is necessary for the SBOL Stack to operate, openrdf-workbench can be useful for debugging purposes.

sudo cp ./openrdf-sesame-4.1.2/war/openrdf-sesame.war /opt/apache-tomcat-8.0.35/webapps/
sudo cp ./openrdf-sesame-4.1.2/war/openrdf-workbench.war /opt/apache-tomcat-8.0.35/webapps/
sudo /opt/apache-tomcat-8.0.35/bin/startup.sh

OpenRDF Sesame should now be accessible at http://localhost:8080/openrdf-sesame, and the workbench at http://localhost:8080/openrdf-workbench.


Installing the Stack API

The latest version of the SBOL Stack API can be obtained using git:

git clone https://github.com/ICO2S/sbolstack-api

To install the dependencies for the SBOL Stack API server, run npm install from the newly cloned directory:

cd sbolstack-api
npm install

The default configuration file for the SBOL Stack is located in config/default.json, and initially points to the triplestore http://localhost:8080/openrdf-sesame and runs the API on port 9090. To start the SBOL Stack API, run:

node server.js

The SBOL Stack should now be accessible at http://localhost:9090/.

4. Configuration

The SBOL Stack has a JSON configuration file located in the config directory of the API software.

  • The port configuration variable defines the port to listen on for incoming API requests.
  • The triplestore configuration variable defines the URL of the OpenRDF Sesame instance, and the name of the default triplestore to use for API requests. The default triplestore will be created automatically if it does not exist.
  • If the SBOL Stack API and OpenRDF Sesame are located on the same server, the url property may point to localhost, where the port depends on the Sesame configuration.
  • The prefixes configuration variable defines a list of URI prefixes to use to allow abbreviation of resource URIs. While defining prefixes is not mandatory, it is recommended to improve readability of RDF/XML serialization.
  • The otherStacks configuration variable defines a list of URLs of other SBOL Stack instances. The data from these instances will automatically be merged into any query to the local instance.

Additionally, the configuration file may contain a users variable with a list of users and their respective permissions. The available permissions are:

  • create — Allow creation of new stores
  • upload — Allow upload of data
  • retrieve — Allow retrieval and searching of data
  • sparql — Allow SPARQL queries

If no permissions are specified for a user, their permissions will default to the anonymousPermissions property.


{
    "port": 9090,

    "triplestore": {
        "url": "http://52.19.236.4:8085/openrdf-sesame",
        "defaultStore": "sbolstack_v1"
    },

    "prefixes": {
        "bacillondex": "http://www.bacillondex.org/"
    },

    "users": [
        {
            "name": "admin",
            "password": "admin",
            "permissions": [
                "create", "upload", "retrieve", "sparql"
            ]
        }
    ],

    "anonymousPermissions": [
        "retrieve", "sparql"
    ]
}

3. RESTful API

For languages without a dedicated client library, the SBOL Stack is accessible through a RESTful HTTP API.

3.1. Upload

POST /store/create

Create a new store in the Stack. This endpoint requires the create permission.

Parameters:

  • storeName — the name of the store to create

POST /

POST /storeName

Upload new data to the Stack, or to a specific named store. This endpoint requires the upload permission.

3.2. SPARQL

GET /sparql

GET /storeName/sparql

Query the stack using SPARQL. The results will be returned as an RDF/XML serialization.

This endpoint requires the sparql permission.

Parameters:

  • query — the SPARQL query to execute

POST /sparql

POST /storeName/sparql

As GET /sparql, but the query is specified as a POST parameter.

This endpoint requires the sparql permission.

Parameters:

  • query — the SPARQL query to execute

3.3. Components

These endpoints are related to ComponentDefinition instances.

Component endpoints of the form /component/uri/* refer to an individual ComponentDefinition, where uri is a fully qualified URI.

GET /component/count

GET /storeName/component/count

Return the total number of ComponentDefinition instances present in the store.

Example: http://synbiohub.org:9090/component/count

GET /component/uri/sbol

GET /storeName/component/uri/sbol

Retrieve an SBOL2 RDF/XML document for the ComponentDefinition. The document will be complete the definition of any referenced URIs, where present in the stack (i.e. all subcomponent definitions and related resources will be included).

Example: http://synbiohub.org:9090/component/http%3A%2F%2Fparts.igem.org%2FPart%3ABBa_S04352/sbol

POST /component/sbol

POST /storeName/component/sbol

As GET /component/uri/sbol, but the component URI is specified as a POST parameter.

Parameters:

  • uri — the URI of the component to retrieve

GET /component/search/sbol

GET /component/search/metadata

GET /storeName/component/search/sbol

GET /storeName/component/search/metadata

Search the stack for ComponentDefinition instances, returning either an SBOL document or JSON metadata describing the component(s).

Parameters (all optional):

  • name — the dcterms:title to search for
  • nameExactMatch — if true, name is searched as a complete title. Otherwise, name is parsed as a wildcard.
  • description — the dcterms:description to search for
  • descriptionExactMatch — if true, description is searched as a complete title. Otherwise, description is parsed as a wildcard.

Example: http://synbiohub.org:9090/component/search?name=BBa*

POST /component/search/template

POST /storeName/component/search/template

Search the stack for ComponentDefinition instances matching an SBOL ComponentDefinition template.

Returns an SBOL2 document with a summary of all matching components.

Parameters:

  • sbol — an RDF/XML SBOL document containing a ComponentDefinition to match.

3.4. Modules

These endpoints are related to ModuleDefinition instances.

Module endpoints of the form /module/uri/* refer to an individual ModuleDefinition, where uri is a fully qualified URI.

GET /module/count

GET /storeName/module/count

Return the total number of ModuleDefinition instances present in the stack.

Example: http://synbiohub.org:9090/module/count

GET /module/uri/sbol

GET /storeName/module/uri/sbol

Retrieve an SBOL2 RDF/XML document for the ModuleDefinition. The document will be complete the definition of any referenced URIs, where present in the stack (i.e. all submodule definitions and related resources will be included).

POST /module/sbol

POST /storeName/module/sbol

As GET /module/uri/sbol, but the module URI is specified as a POST parameter.

Parameters:

  • uri — the URI of the module to retrieve

GET /module/search/sbol

GET /module/search/metadata

  • descriptionExactMatch — if true, description is searched as a complete title. Otherwise, description is parsed as a wildcard.
